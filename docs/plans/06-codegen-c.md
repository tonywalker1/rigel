# Plan 06 — C Code Generation

**Status:** Elaborated (designed). Implementation deferred — interpreter (Plan 09) is the
priority for Slice 3.

## Purpose

Transforms checked IR into C source code. This is the compilation backend: monomorphized,
tail-call-optimized C with checked arithmetic and RAII scope cleanup. Designed here in full
so that implementation can proceed without further planning.

- §6 (Compilation to C) — translation strategy, tail calls, checked arithmetic
- §5 (Memory Model) — ownership, RAII, ref counting in generated C
- §2.2 (Qualifiers) — `unchecked` disables overflow checks, `unsigned` maps to `uint*_t`

## Interface Contract

### Source: `src/rigel/codegen.py`

**Entry point:**
```python
def codegen(ir: list[TypedExpr], *, module_name: str = "main") -> str:
    """Emit C source code from checked IR.

    Returns a string containing the complete C translation unit.
    Raises CodegenError on untranslatable constructs.
    """
```

**Error class** (added to `common.py`):
```python
class CodegenError(RigelError):
    """Error during C code generation."""
```

### Output structure

The generated C file has this layout:
```c
/* Generated by Rigel compiler — do not edit */
#include "rigel_rt.h"
#include <stdint.h>
#include <stdbool.h>
#include <stdio.h>

/* Forward declarations */
...

/* Function definitions */
...

/* Main entry point */
int main(void) {
    /* top-level expressions */
    ...
    return 0;
}
```

## Behavioral Requirements

### Literals

- Integer literals emit as C integer constants with appropriate type: `(int64_t)42`.
- Float literals emit as C float constants: `3.14` (double) or `3.14f` (float).
- String literals emit as calls to `rigel_string_from_cstr("...")`.
- Bool literals emit as `true`/`false`.

### Bindings

- `let` emits a local variable declaration: `int64_t x = 42;`.
- `set` emits an assignment: `x = 43;`.
- Immutable bindings emit `const`: `const int64_t x = 42;`.

### Checked Arithmetic

Default (no `unchecked` qualifier):
```c
int64_t rigel_add_i64(int64_t a, int64_t b) {
    int64_t result;
    if (__builtin_add_overflow(a, b, &result)) {
        rigel_panic("integer overflow");
    }
    return result;
}
```

With `unchecked` qualifier: emit plain C operators (`a + b`).

Division by zero always checked (even `unchecked` — it's UB in C):
```c
if (b == 0) rigel_panic("division by zero");
```

### Control Flow

- `if` → C `if`/`else`. Two-arm if is an expression via comma operator or temp variable.
- `cond` → chained `if`/`else if`/`else`.
- `match` → `switch` for integer patterns; chained `if`/`else` for general patterns.
- `do` → compound statement `{ stmt1; stmt2; ...; result = stmtN; }`.

### Functions

- Named lambdas (let-bound) emit as C functions.
- Anonymous lambdas or lambdas with captures emit as `rigel_closure_t` (runtime struct with
  function pointer + captured environment).
- Function calls: direct call for known functions, indirect call via closure dispatch for
  higher-order functions.

### Tail Call Optimization

Tail-recursive calls (self-call in tail position) are transformed to loops:
```c
int64_t fact(int64_t n) {
    TAIL_CALL:
    if (n == 0) return 1;
    /* Tail call: fact(n - 1) * n — NOT tail position, so no optimization here */
    return n * fact(n - 1);
}
```

Only self-tail-calls in direct tail position are optimized. Mutual recursion and non-tail
calls are emitted as regular function calls.

### RAII / Scope Cleanup

Ref-counted values (strings, closures) need cleanup at scope exit:
```c
{
    rigel_string_t s = rigel_string_from_cstr("hello");
    /* ... use s ... */
    rigel_string_release(s);  /* scope exit cleanup */
}
```

The `:release` keyword section on type definitions maps to a cleanup function called at scope
exit. Codegen inserts release calls before every `return` and before effect raises.

### Effects

- `raise` → call to `rigel_effect_raise(effect_id, args...)` which uses `longjmp`.
- `handle` → `setjmp`/`longjmp` frame:
  ```c
  rigel_effect_frame_t frame;
  if (setjmp(frame.buf) == 0) {
      rigel_effect_push(&frame);
      /* body */
      rigel_effect_pop();
  } else {
      /* handler dispatch based on frame.effect_id */
  }
  ```

## Monomorphization Strategy

Generic functions (when implemented) are instantiated at each concrete type used at a call site.
Each instantiation produces a separate C function:
```c
/* (let add (lambda (:args (a T) (b T)) (:returns T) (+ a b))) */
/* Called as (add 1:int32 2:int32) and (add 1.0 2.0): */
int32_t add_int32(int32_t a, int32_t b) { return rigel_add_i32(a, b); }
double add_float64(double a, double b) { return a + b; }
```

Name mangling: `<function>_<type1>_<type2>_...`.

## Design Notes

### Why defer codegen?

The interpreter provides immediate feedback and enables the full test suite to run. Codegen
requires a runtime library (Plan 07), a C compiler in the toolchain, and careful handling of
memory management. Building the interpreter first validates the language semantics; codegen
then becomes a translation exercise against known-correct behavior.

### Why setjmp/longjmp for effects?

The simplest C mechanism for non-local control flow. Alternatives (continuation-passing,
explicit stack manipulation) are more complex and not needed for non-resumable effects.
If resumable effects are added later, a continuation-based approach would replace this.

### Division semantics

Codegen should use C's truncation-toward-zero semantics for integer division (matching the C
target), even though the interpreter uses Python's floor division. This discrepancy is
intentional for now — the spec should eventually settle on one behavior.

## Regeneration Instructions

- Generate `src/rigel/codegen.py`.
- Import checked IR nodes from `rigel.check`, types from `rigel.types`, errors from
  `rigel.common`.
- A `CEmitter` class accumulates C source fragments and manages indentation.
- Main dispatch via `match` statement on `TypedExpr` subtypes, similar to the interpreter.
- Forward declarations pass: collect all named functions, emit prototypes.
- Main pass: emit function definitions and top-level code inside `main()`.
- String escaping: C string literals must escape `\`, `"`, `\n`, `\t`, etc.
